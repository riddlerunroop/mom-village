- `supabase/migration_13_monthly_chart_seed.sql` — widens the `section` check constraint to the 6 real category keys, makes `title` nullable (the locked content is written as flowing sentences, not short-title-plus-elaboration pairs, so the card UI now renders each item as a single paragraph rather than inventing an artificial title), clears any stale rows, and inserts all 1,031 real content rows. **Not yet run — this is the one manual step left for Roop, same pattern as every other migration in this project: paste into Supabase's SQL editor and run.**

## Birth-date prompt — built 2026-07-24 (night)

Roop's question: how does the app know a pregnant mother has actually delivered, if she doesn't log in again for days or weeks after? Until she updates her profile, the app keeps counting from her due date, showing pregnancy content to a mother who's already given birth. Confirmed fix, her framing: a hard-to-miss prompt that "starts flashing the moment she enters 9th month," reachable whenever she next opens the app after delivery, however late that is.

- `shouldPromptBirth(dueDate, babyDob)` added to `src/lib/monthCalculator.ts` — true whenever `baby_dob` isn't set yet and she's reached `calculateMonthNumber(dueDate) >= -1` (final pregnancy month or overdue). Deliberately keeps returning true on every load until she actually confirms the birth, rather than firing once — she might not log back in until well after delivery.
- `src/components/DashboardNav.tsx` — now takes an optional `promptBirth` prop; when true, renders a small pulsing terracotta dot (ping animation) on the "Monthly chart" tab.
- `src/app/dashboard/layout.tsx` — computes `promptBirth` from the profile's `due_date`/`baby_dob` (already fetched for the existing redirect checks) and passes it to `DashboardNav`.
- `src/app/dashboard/page.tsx` — shows a prominent terracotta-bordered banner ("Has your baby arrived?") above the month header when `promptBirth` is true, linking to the new confirm-birth flow.
- `src/app/dashboard/confirm-birth/page.tsx` + `ConfirmBirthClient.tsx` — new short-form page: birth date + delivery type (normal/c-section, same options as onboarding). On save, updates `profiles.baby_dob`/`delivery_type` and redirects to `/dashboard` — the layout's existing `birth_welcome_seen` redirect chain then automatically takes her to the welcome-baby moment, so this flow doesn't duplicate any of that logic, it just supplies the missing trigger.

No new Supabase migration needed (uses existing `due_date`/`baby_dob`/`delivery_type` columns). Verified clean on `tsc`/`eslint`. **Not yet deployed** — needs a GitHub Desktop commit + push.

Important technical note for future sessions: making pages "flip" requires the book to use **fixed-size virtual pages** (content is pre-paginated into fixed chunks, not a responsive scroll) — the whole book then scales as a unit to fit the screen via `size="stretch"` with min/max bounds, so page breaks stay consistent across devices even though the physical page looks bigger on desktop than on a phone.

**Prototype built with one book first, as agreed — "Money, Understood" (the shortest of the six, ~17,000 words).** Roop explicitly asked to see/approve this before the other five get converted the same way, given it's a real format decision. Pipeline built:

- `docx → HTML (pandoc) → parsed content blocks → paginated JSON`, done as an offline conversion (not at runtime). The HTML parse (via BeautifulSoup) walks the manuscript and classifies each element into one of five block types: paragraph, heading, list, "box" (the manuscript's single-cell callout tables like REMEMBER THIS / TRY THIS TODAY — rendered as styled callout cards), or "table" (real multi-column data tables, e.g. the product-comparison and resource-directory tables — large tables are automatically split into smaller chunks of ~3 data rows each so they never overflow a single page).
- A greedy pagination pass packs these blocks onto virtual pages using a weighted character budget (~620 weighted characters/page, with headings/boxes/tables weighted heavier than their raw character count to account for the extra visual space they take). Money, Understood paginated to **306 pages** at this budget — every chapter always starts a fresh page.
- Output stored as static JSON at `src/content/library/money-understood.json` (not a database table — book content is fixed editorial content, not user data, so no migration was needed for this).
- Each book's actual cover image (already embedded in its locked manuscript docx) was extracted directly from the docx's `word/media/` folder for reuse — all six cover PNGs now live in `public/library-covers/`, used both on the reader's front cover page and the catalog thumbnails.

**Page-turn sound effect — synthesized, not sourced.** Rather than risk a copyrighted/licensed sound clip, a short (~0.3s) "swish" sound was synthesized programmatically in Python (filtered white noise with a fast-attack/quick-decay envelope, plus a very brief low thump at the very start to suggest the page corner catching) and encoded to `public/sounds/page-turn.mp3` (~2.5KB).
